# VR-Based Biofeedback Acrophobia Therapy System

A closed-loop virtual reality therapy system for treating acrophobia (fear of heights) using real-time biofeedback. The system monitors the patient's heart rate (PPG) and galvanic skin response (GSR) during VR exposure therapy, adapts the session dynamically based on physiological state, and generates structured JSON session logs for post-session clinical review.

## Overview

This project combines embedded hardware, a VR application, and a clinical analytics dashboard into a single therapy pipeline:

1. An **ESP32 sensor node** reads PPG and GSR data in real time, processes the signals on-device, and broadcasts them over MQTT via a self-hosted Wi-Fi access point.
2. A **Unity VR application** connects to the MQTT broker, reads the biofeedback stream, and adapts the exposure therapy session — progressing levels, triggering regression, or returning to relaxation based on the patient's stress state.
3. A **Streamlit clinical dashboard** loads the JSON session logs generated by Unity and provides therapists with charts, behavioral metrics, and session scoring.

## Hardware Setup

### Components

| Component | Part | GPIO |
|---|---|---|
| Microcontroller | ESP32 DevKit | — |
| PPG Sensor | Analog pulse sensor | 34 |
| GSR Sensor | Analog conductance sensor | 35 |
| OLED Display | SH1106 128×64 (I2C) | SDA 21, SCL 22 |
| Blue LED | Beat indicator | 25 |
| Green LED | System ready | 26 |
| Red LED | Alert / stress | 27 |
| Start Button | Session start | 14 |

### Wiring Notes

- OLED uses I2C at address `0x3C`, 100 kHz clock for stability.
- ADC attenuation set to `ADC_11db` for the full 0–3.3V range on both analog pins.
- The ESP32 runs as a **Wi-Fi Access Point** (`ESP32_AP` / `87654321`) and hosts its own **MQTT broker on port 1883** — no external router needed.

### Signal Processing Pipeline (on-device)

```
Raw ADC → Moving Average (n=4) → IIR High-Pass (DC removal, α=0.97)
       → Amplify (gain=8) → Threshold peak detection → BPM calculation
       → Rolling average buffer (n=5) → Trend confirmation filter → lastValidBPM
```

GSR is baseline-calibrated at startup over 500 samples (~2.5s). Stress is flagged when conductance rises more than 15% above baseline, confirmed by a second-pass reading.

### MQTT Topics

| Topic | Direction | Payload |
|---|---|---|
| `sensors/bpm` | ESP32 → Unity | Integer BPM value |
| `sensors/gsr` | ESP32 → Unity | `"STRESSED"` or `"RELAXED/NORMAL"` |
| `sensors/status` | ESP32 → Unity | Status string |
| `patient/alert` | Unity → ESP32 | `"ALERT"` or `"NORMAL"` |

## Unity Project

### Requirements

- Unity 2022.3 LTS or newer
- XR Interaction Toolkit
- M2Mqtt Unity package
- TextMeshPro
- Unity Input System

### Key Scripts

**`BiofeedbackMQTT.cs`**
Central data hub. Connects to the ESP32 MQTT broker, decodes incoming BPM and GSR payloads, and logs all biofeedback samples to a structured `SessionData` object. Falls back to a keyboard-driven **Dummy Mode** automatically if the broker is unreachable. Saves a timestamped JSON session file to `Application.persistentDataPath` on quit.

**`BaselineManager.cs`**
Singleton that persists across scenes. Collects BPM and GSR samples over a timed window before therapy begins to establish the patient's resting baseline. Used by the progression manager and companion NPC as the reference for stress thresholds. On subsequent calls (after a panic event), treats the collection window as a recovery relaxation phase instead.

**`SceneProgressionManager.cs`**
The core therapy loop. Implements a coroutine-driven state machine that:
- Starts at Level 0 (safe neutral environment)
- Waits for the patient to stabilise, then prompts for progression
- Lets the patient choose Indoor or Outdoor exposure path
- Monitors BPM continuously against elevated (20% above baseline) and panic (40% above baseline) thresholds
- Advances levels on stabilisation, regresses on sustained elevation, and returns to relaxation on true panic
- Logs every transition, prompt, and threshold event via `BiofeedbackMQTT`

**`NPC.cs` (CompanionNPC)**
An AI voice companion that follows the patient at Level 0 and speaks contextually based on their stress state. Plays level-entry lines, panic detection responses, recovery encouragement, and avoidance nudges. Uses a no-repeat random line selector per state and drains an essential-line queue to ensure critical messages are never skipped.

### Session JSON Format

Sessions are saved automatically to the device's persistent data path as:
```
session_YYYY-MM-DD_HH-mm-ss.json
```

The file contains three arrays:
- `baselineMetrics` — samples collected during the resting baseline phase
- `metrics` — samples collected during all exposure phases
- `events` — timestamped event log (level transitions, prompts, safe zone entries, panic detections, etc.)

## Clinical Dashboard

A Streamlit web app for therapists to review session data after each therapy session.

### Running the Dashboard

```bash
cd ClinicalDashboard
pip install -r requirements.txt
streamlit run app.py
```

Then open `http://localhost:8501` in a browser and upload a session JSON file.

### Dashboard Sections

| Tab | Content |
|---|---|
| Baseline Signals | BPM time series during the baseline collection window |
| Exposure Signals | BPM over time with baseline/elevated/panic threshold lines and event markers |
| Level Analysis | Average and max BPM per level, time spent per level, stabilisation times |
| GSR Summary | GSR state distribution across the session and per level |
| Events | Full chronological event log table and event frequency bar chart |
| Raw Tables | Raw baseline, exposure, and event DataFrames |

### Computed Metrics

- **Therapy Performance Score** (0–100) — composite score based on levels completed, stress ratios, safe zone time, and BPM trend
- **Session Status** — Stable 🟢 / Elevated 🟡 / High Stress 🔴
- **Avg Decision Time** — time between a proceed prompt appearing and the patient pressing the button
- **Time in Safe Zone** — total seconds spent avoiding the exposure stimulus
- **Elevated / High Stress %** — proportion of exposure samples above each BPM threshold
- **BPM Trend Delta** — change in average BPM from the first 20% to the last 20% of the session

## Getting Started

1. **Flash the ESP32** — open `ESP32BiofeedbackCode/FullHardware.ino` in Arduino IDE, install the required libraries, select your ESP32 board, and upload.

2. **Wire the hardware** — connect the PPG sensor to GPIO 34, GSR sensor to GPIO 35, and OLED to SDA 21 / SCL 22. Connect LEDs and the start button as listed in the hardware table above.

3. **Power on the ESP32** — attach the sensors to the patient, then press the start button. The OLED will show a splash screen and begin GSR calibration automatically.

4. **Open the Unity project** — import the scripts from `UnityScripts/` into your Unity project. Connect your VR headset, ensure it is on the same Wi-Fi network as the ESP32 AP (`ESP32_AP`), and set the MQTT broker IP in `BiofeedbackMQTT` to `192.168.4.1`.

5. **Run the VR session** — the Unity app will automatically connect to the ESP32 broker or fall back to Dummy Mode (keyboard-controlled BPM) if unreachable. The session JSON is saved on application quit.

6. **Review with the dashboard** — copy the session JSON from `Application.persistentDataPath` to the machine running the dashboard, launch Streamlit, and upload the file.
